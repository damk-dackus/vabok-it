# Structuur Software Tester Opleiding

## Inleiding

Welkom bij de Software Tester opleiding! Deze module leert je de fundamenten van software testen en hoe je professionele testdocumentatie maakt. Software testen is een essentieel onderdeel van softwareontwikkeling - het zorgt ervoor dat software werkt zoals bedoeld, gebruiksvriendelijk is en betrouwbaar functioneert.

In deze praktijkgerichte module ga je stap voor stap leren hoe je software systematisch test. Je werkt met de website **IT Showcase** als praktijkvoorbeeld, zodat je direct ervaring opdoet met het testen van een echte applicatie. Door het doorlopen van deze module ontwikkel je niet alleen technische vaardigheden, maar ook het vermogen om kritisch te kijken naar software en verbetervoorstellen te formuleren.

Na het voltooien van deze module ben je in staat om zelfstandig testplannen te maken, testen uit te voeren, testrapporten te schrijven en verbetervoorstellen te presenteren. Deze vaardigheden zijn onmisbaar in je toekomstige carrière als IT-professional, of je nu gaat werken als tester, ontwikkelaar of in een andere rol binnen de software branche.

## Sessie 2: Software Requirement Specification (SRS)

### Les onderdeel
- Wat is een Software Requirement Specification (SRS)?
- Waarom zijn goede eisen belangrijk?
- Hoe ziet een SRS document eruit?
- Hoe schrijf je eisen op?
- Goede manieren om een SRS te maken

### Opdracht
**Deelopdracht 2: User stories, backlog en definition of done**

**Doel**
Je leert hoe je nieuwe wensen van een klant vertaalt naar user stories en deze verwerkt in een bestaande Software Requirement Specification (SRS). Daarnaast leer je hoe je een backlog opstelt en een definition of done maakt.

**Beschrijving**
De klant is erg blij met het voorlopige ontwerp, maar heeft twee nieuwe wensen doorgegeven die nog niet in het ontwerp stonden.

1. **Embedded code om te delen**: De klant wil dat bezoekers embedbare code kunnen creëren en delen, zodat een project elders getoond kan worden. In de embedded weergave speelt een kleine presentatie af met screenshots van het project, de titel en de korte omschrijving.
2. **Logboek voor een project**: De klant wil dat er per project een logboek is waarin voortgang, wijzigingen of notities kunnen worden bijgehouden.

Aan jou de taak om deze nieuwe wensen te vertalen naar user stories en ze toe te voegen aan het bestaande SRS document. Daarna maak je een backlog voor het hele project.

**Stappen**
- **Stap 1: Lees het bestaande SRS document**
  - Pak het SRS document erbij
  - Lees de bestaande functionele eisen door om te zien hoe ze zijn opgeschreven

- **Stap 2: Vertaal de nieuwe wensen naar user stories**
  - Maak een user story voor "Embedded code om te delen". Formuleer deze met:
    - Wie: (bijv. bezoeker of student)
    - Wat: (bijv. embedbare code creëren en delen, waarin een kleine presentatie met screenshots, titel en korte omschrijving afspeelt)
    - Waarom: (bijv. project presenteren op eigen website of portfolio)
  - Maak een user story voor "Logboek voor een project". Formuleer deze ook volgens het wie-wat-waarom format.
  - Zorg dat beide user stories realistisch en duidelijk zijn.

- **Stap 3: Verwerk de user stories in de SRS**
  - Voeg de twee nieuwe user stories toe aan het SRS document
  - Geef ze een logisch nummer (bijv. FE10.1 en FE10.2 of een nieuwe categorie)
  - Zorg dat ze passen in de stijl van het document

- **Stap 4: Maak een definition of done**
  - Bedenk wanneer een user story klaar is
  - Schrijf op wat er moet gebeuren voordat een user story "done" is (bijv. getest, documentatie, code review)
  - Maak een lijst met criteria voor "done"

- **Stap 5: Maak een backlog**
  - Maak een overzicht (backlog) met daarin de nieuwe user stories én minimaal 5 andere belangrijke user stories uit het SRS
  - Zet bij elke user story:
    - Nummer
    - Wie (gebruiker)
    - Wat (functionaliteit)
    - Waarom (reden)
  - Voeg de definition of done toe aan dit document

- **Stap 6: Stem af met stakeholders**
  - Controleer of de nieuwe user stories duidelijk zijn voor de ontwikkelaars
  - Vraag feedback aan je "opdrachtgever" (docent/begeleider) over de formulering

**Op te leveren**
- Een aangepast SRS document (of een kopie ervan) waarin de twee nieuwe user stories zijn verwerkt
- Een backlog document met daarin de nieuwe user stories, een aantal bestaande stories en de definition of done

**Beoordeling/toetsing**
De beoordeling vindt plaats middels voltooid/niet voltooid.

**Beoordelingsinstrument**
Wanneer heb je de opdracht behaald?
- Zijn de twee nieuwe wensen correct vertaald naar user stories?
- Zijn de user stories correct toegevoegd aan het SRS document?
- Is de backlog en definition of done compleet en duidelijk?

### Opdracht
**Deelopdracht 3: Niet-functionele eisen verifiëren in de applicatie**

**Doel**

## Sessie 9: Prioritiseren van de verbetervoorstellen

### Les onderdeel
- Hoe bepaal je wat eerst moet?
- Belangrijkheid versus moeite
- MoSCoW methode
- Prioriteit op basis van risico's
- Planning maken

### Opdracht
**Deelopdracht 9: Verbetervoorstellen prioriteren en plannen voor de IT Showcase**

**Doel**
Je leert hoe je verbetervoorstellen prioriteert met de MoSCoW methode voor de IT Showcase. Je bepaalt wat eerst moet gebeuren en maakt een planning. Je schat tijd in en bepaalt wat je nodig hebt voor elk voorstel.

**Beschrijving**
Voor deze opdracht bepaal je wat eerst moet gebeuren met je verbetervoorstellen voor de IT Showcase. Je gebruikt de MoSCoW methode om prioriteiten te stellen. Je maakt een planning van wanneer je wat gaat doen. Je schat tijd in en bepaalt wat je nodig hebt voor elk voorstel, rekening houdend met de verschillende gebruikersrollen en functionaliteiten van de IT Showcase.

**Stappen**
- **Stap 1: Pak je verbetervoorstellen erbij**
  - Gebruik het document met verbetervoorstellen uit sessie 8
  - Lees alle voorstellen nog een keer door
  - Zorg dat je begrijpt wat elk voorstel inhoudt en welke functionaliteit van de IT Showcase dit betreft

- **Stap 2: Gebruik de MoSCoW methode**
  - Verdeel alle voorstellen in 4 groepen:
    - **Must have**: Moet absoluut gebeuren (zeer belangrijk, bijvoorbeeld: kritieke fouten, veiligheidsproblemen)
    - **Should have**: Zou moeten gebeuren (belangrijk, bijvoorbeeld: belangrijke gebruiksvriendelijkheid verbeteringen)
    - **Could have**: Zou kunnen gebeuren (minder belangrijk, bijvoorbeeld: nice-to-have features)
    - **Won't have**: Gaat nu niet gebeuren (laagste prioriteit, bijvoorbeeld: toekomstige wensen)
  - Schrijf bij elk voorstel in welke groep het hoort
  - Leg uit waarom je het in die groep hebt gezet, rekening houdend met de impact op gebruikers van de IT Showcase

- **Stap 3: Bepaal de volgorde binnen elke groep**
  - Binnen de "Must have" groep: wat moet het eerst? (bijvoorbeeld: authenticatie fouten voor projectbeheer fouten)
  - Binnen de "Should have" groep: wat is het belangrijkst?
  - Bepaal de volgorde op basis van:
    - Hoe belangrijk het is voor de IT Showcase en haar gebruikers
    - Hoeveel tijd het kost om te implementeren
    - Wat er eerst moet gebeuren voordat je iets anders kunt doen (afhankelijkheden)
    - Welke gebruikersrol het meest wordt beïnvloed

- **Stap 4: Schat de tijd in**
  - Voor elk voorstel: schat in hoeveel tijd het kost om te implementeren
  - Denk na over: uren, dagen, of weken
  - Houd rekening met de complexiteit van de IT Showcase
  - Schrijf bij elk voorstel hoeveel tijd je denkt dat het kost

- **Stap 5: Bepaal wat je nodig hebt**
  - Voor elk voorstel: wat heb je nodig om het te implementeren?
  - Denk aan: ontwikkelaars, designers, testers, tools, toegang tot bepaalde delen van de IT Showcase
  - Schrijf op wat je nodig hebt voor elk voorstel

- **Stap 6: Maak een planning**
  - Maak een planning per week of per maand
  - Zet de "Must have" voorstellen eerst in de planning
  - Zet daarna de "Should have" voorstellen
  - Zet als laatste de "Could have" voorstellen
  - Zorg dat de planning realistisch is voor de IT Showcase
  - Houd rekening met afhankelijkheden tussen voorstellen

- **Stap 7: Maak een overzicht**
  - Maak een document met:
    - Alle voorstellen met hun prioriteit (MoSCoW)
    - De volgorde waarin je ze gaat doen
    - Geschatte tijd per voorstel
    - Wat je nodig hebt
    - Een planning wanneer je wat doet
    - Welke functionaliteit van de IT Showcase elk voorstel betreft

**Op te leveren**
- Een prioriterings- en planningsdocument (minimaal 2 pagina's)
- Alle voorstellen moeten een MoSCoW prioriteit hebben
- Er moet een duidelijke planning in staan specifiek voor de IT Showcase

#### Opdracht 3: Testen
